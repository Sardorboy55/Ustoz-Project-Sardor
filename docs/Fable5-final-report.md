## 9. Booking flow (как работает)

Гость: профиль → виджет → «Войти и забронировать» → `/auth?next=` → возврат. Ученик: предмет →
длительность (или бесплатный пробный 20 мин) → день/слот из `get_free_slots` → summary с политикой
отмены → `booking-create` (цена считается сервером, лимит FREE, exclusion-защита от двойного
бронирования) → success. `pending_payment` истекает за 15 мин (pg_cron) — UI честно это объясняет.
Пробный — сразу `paid`. Отмена: ≥12 ч — 100 % на внутренний баланс, <12 ч — 0, преподавателем —
всегда 100 % ученику + страйк. Проверено E2E на живом стеке (создание + отмена).

## 17. Риски перед production

1. **Координация с `phase-4-payments`**: при мерже возможны конфликты в `booking_sheet.dart`,
   `router.dart`, messages/ARB; фикс PGRST201 в `booking-create` обязан попасть в результат мержа.
   Если Codex сделает `supabase db reset` из `C:\ustoz` до мержа Fable5 — таблицы `student_favorites`
   не будет и UI избранного станет падать (и наоборот: локальный стек теперь смонтирован из
   `C:\ustoz-fable5`; перезапустить `supabase stop/start` из нужной папки).
2. Неизвестные URL отдают HTTP 200 с noindex-404-контентом (стриминг Next 16) — для краулеров
   безопасно, для строгого 404 нужно отдельное решение.
3. Чаты: непрочитанность не считается (в `messages` нет read-маркера) — кандидат на аддитивную миграцию.
4. «Скрыть» в модерации меняет статус очереди, но само сообщение в чате остаётся (нет `is_hidden` у messages).
5. Демо-данные seed обязаны быть вычищены до прода (помечены комментариями).
6. Урок пока не проводится внутри платформы (видеокомната — Фаза 5): FAQ честно говорит об этом.
7. Тестов на web/admin нет (lint+build+smoke руками); e2e букинга — `scripts/test_booking_e2e.sh`.
8. Прод-Supabase: применить миграции, включить pg_cron, настроить sms-hook secret.
