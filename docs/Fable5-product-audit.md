# Fable5 — Продуктовый аудит USTOZ

> Ветка: `Fable5` (от `main`, фазы 0–3 смержены). Дата: 2026-06-12.
> Цель ветки: довести продукт от «работающего скелета» до уровня готового MVP-маркетплейса (ориентиры: italki, Preply, Booksy) без изменения архитектуры и без выхода в зону владельца (платежи, ключи, юр.доки).

---

## 1. Что уже работает (проверено по коду)

### Backend (Supabase) — самая зрелая часть проекта
- **Полная схема БД** (14 миграций): profiles, teacher_profiles, categories/subjects, teacher_subjects (цены 30/60/90 в тийинах, пробные, скидки пакетов), availability_rules/exceptions, bookings (GiST exclusion против двойного бронирования), lessons, payments, wallets + wallet_transactions, student_balance_transactions, payout_requests, **reviews**, **chats/messages** (с маскировкой контактов триггером), notifications, gamification/xp_events, app_settings, admin_audit_log, moderation_queue, support_tickets.
- **RLS на каждой таблице**, узкие GRANT'ы, деньги только через SECURITY DEFINER-функции.
- **RPC**: `catalog_teachers` (поиск/фильтры/сортировка с формулой ранжирования), `get_free_slots`, `become_teacher`, `wallet_request_payout`, `wallet_resolve_payout` (admin), `admin_adjust_student_balance` (admin), `grant_xp`, `expire_stale_bookings` (pg_cron каждые 5 мин).
- **Edge Functions**: `booking-create` (цена считается сервером, лимит FREE, бесплатный пробный 1 на пару, напоминания в notifications), `booking-cancel` (политика ≥12ч/100% на баланс, страйки преподавателю), `sms-hook` (mock + Eskiz-ветка).
- **Seed**: 8 категорий, 30+ предметов, 5 демо-преподавателей с ценами и расписанием, админ, ученик, app_settings.
- **Storage**: avatars, intro-videos (публичные), chat-files, homework (приватные) с политиками.
- **Realtime**: messages, chats, whiteboard_strokes, bookings, notifications.

### Mobile (Flutter) — функционально, но визуально «Фаза 0»
- Auth: телефон → OTP (60с resend), онбординг 3 слайда, setup имя+интересы.
- Каталог: debounce-поиск, чипы категорий, сортировка, фильтр «пробный», карточки.
- Профиль преподавателя: статистика, био, предметы с ценами, кнопка брони.
- Бронирование: bottom-sheet (предмет → длительность/пробный → день → слот), обработка SLOT_TAKEN/TRIAL_USED/TEACHER_LIMIT.
- «Мои уроки»: вкладки предстоящие/прошлые, переключатель роли, отмена с подтверждением.
- Кабинет преподавателя: анкета (uz/ru, фото/видео загрузка), предметы/цены, редактор расписания + исключения.

### Web (Next.js 16) — публичная часть в порядке, продуктовая отсутствует
- Каталог `/catalog`: SSR, фильтры GET-формой, пагинация.
- Профиль `/t/[slug]`: ISR 60с, schema.org Person/AggregateRating/Offer, OG, hreflang (Lighthouse SEO 92).
- `/s/[subject]`, sitemap c hreflang, robots.
- Auth по телефону, мини-кабинет (имя, «стать преподавателем», предметы+цены).
- i18n: next-intl uz (default) / ru, покрытие полное для существующих экранов.

### Admin — есть только каркас (1 страница со списком будущих разделов).

## 3. Незавершённые пользовательские сценарии

1. **Ученик на web не может забронировать вообще** — главный разрыв: SEO приводит на профиль, дальше тупик.
2. После бронирования (mobile) нет экрана успеха/чекаута — сразу выброс в «Мои уроки»; оплата не объяснена.
3. «Написать преподавателю» — нет нигде (чат-таблицы и Realtime готовы).
4. Оставить отзыв после завершённого урока — нет UI (RLS уже разрешает).
5. Уведомления пишутся в БД (`booking-create` создаёт напоминания), но ни один клиент их не показывает.
6. Гость на web: бронирование должно вести в login с возвратом — нет.
7. Кабинет преподавателя на web сильно беднее мобильного (нет анкеты/фото/видео/расписания).
8. Дашборд преподавателя (ближайшие уроки, доход) — нет ни на web, ни в mobile.

## 4. Какие экраны нужно улучшить (приоритет)

**Web:** лендинг (полный) → профиль преподавателя (отзывы, календарь, FAQ, похожие) → бронирование + mock-checkout → каталог (карточки, фильтры-UX, skeleton) → кабинет-дашборд → «Стать преподавателем» → сообщения → error/loading/404.

**Mobile:** тема + дизайн-токены → bottom nav (5 табов) → главная (приветствие, поиск, категории, ближайший урок, популярные, баннер) → профиль преподавателя (отзывы, календарь) → success-экран брони + checkout-заглушка → чаты (живые) → уведомления → отзыв после урока → избранное.

**Admin:** вход с проверкой `is_admin` → layout → дашборд (метрики) → Users/Teachers/Bookings/Payments/Payouts/Reviews/Categories/Settings/Support.

## 5. Какие API уже есть (для фронтов)

- `rpc catalog_teachers(...)` — каталог с total_count.
- `rpc get_free_slots(teacher_id, from, to, duration)` — слоты (SECURITY DEFINER, виден аноним).
- `functions/booking-create`, `functions/booking-cancel` — брони.
- `rpc become_teacher()`, `rpc wallet_request_payout`, `rpc wallet_resolve_payout` (admin), `rpc admin_adjust_student_balance` (admin).
- Прямые select'ы под RLS: bookings (участники+админ), reviews (публичные), chats/messages (участники, insert разрешён), notifications (свои, update read_at), wallets/transactions (свои+админ), payments (свои+админ), gamification (своя).
- Realtime: messages, chats, bookings, notifications.

## 6. Что можно доделать БЕЗ владельца (план этой ветки)

- Весь UI/UX web + mobile + admin (см. §4).
- **Живые** чаты, отзывы, уведомления, избранное — бэкенд уже позволяет (favorites — одна аддитивная миграция).
- Mock-checkout: заказ-summary + плашки Payme/Click/Uzum «подключит владелец» + честный статус `pending_payment` (TTL 15 мин — реальное поведение системы).
- Аддитивные миграции: `student_favorites`, недостающие admin-write-политики (бан, скрытие отзывов, CRUD категорий, app_settings, support).
- Seed: больше демо-учеников, completed-брони и отзывы — чтобы профили выглядели живыми.
- SEO-доводка, error/loading states, формы с валидацией, i18n новых строк uz/ru.

## 7. Что требует владельца (НЕ делаем здесь)

Платёжные интеграции (идут в ветке `phase-4-payments` — не трогаем), ключи Payme/Click/Uzum/Eskiz/Agora/FCM, домен, логотип/брендинг, юр.документы, реальные цены/категории/преподаватели, прод-Supabase. Полный список — в финальном отчёте.

## 8. Риски перед запуском

1. **RLS-пробел админки**: у админа нет прав на запись (бан, категории, настройки) — закрываем аддитивной миграцией с `is_admin()`-политиками (через RPC уже доступны выплаты и баланс).
2. `pending_payment` истекает за 15 мин — до подключения оплат web-бронь без бесплатного пробного «сгорает»; это честное поведение, но в UI нужно явно объяснять (делаем).
3. Контент демо-данных в проде надо заменить (помечен как seed).
4. Phone OTP в mock-режиме — до Eskiz-договора вход в прод невозможен.
5. Видеокомната (Фаза 5) — урок пока не проводится внутри платформы; в UI занятий это must-explain.
6. Незакоммиченная работа в `phase-4-payments` (checkout, вебхуки) — при мерже Fable5 возможны конфликты в `booking_sheet.dart`, `router.dart`, messages-файлах; разводить аккуратно (Fable5 не трогает платёжные файлы).
7. apps/web и admin не имеют тестов; mobile — только базовые. Минимум для прода: e2e бронирования (скрипт уже есть) + smoke web.
